# Deskflow-Plus 技术架构文档

## 1. 技术栈

核心推荐技术栈：

- Rust
- Tauri 2
- Tokio
- Rustls
- SQLite
- Serde
- tracing

选择原则：

- 核心逻辑统一在 Rust 中实现
- UI 与高频输入链路隔离
- 网络、安全、配置和平台逻辑采用清晰模块边界
- MVP 优先选择复杂度可控、容易验证的实现

## 2. 总体架构

Deskflow-Plus 采用“桌面壳 + 后台核心进程”的架构，而不是把所有逻辑塞进单一 UI 线程：

- `app-desktop`：Tauri 桌面壳，负责设置界面、托盘/菜单栏、配对确认、权限引导、日志展示
- `core-service`：Rust 后台核心进程，负责输入、会话、安全、拓扑、剪贴板等实时逻辑
- `local-ipc`：桌面壳与核心进程之间的本地 IPC，用于配置下发、状态回传、用户确认和诊断

架构目标：

- GUI 不参与高频输入实时链
- 核心输入、网络和安全逻辑由 Rust 核心统一管理
- UI 仅负责配置、状态展示、确认交互和任务入口
- MVP 不引入 Windows 特权服务或 macOS 登录前助手

## 3. 模块划分

### `core-protocol`

负责：

- 协议版本协商
- 帧格式与序列化
- 配对消息
- 会话控制消息
- 输入事件消息
- 剪贴板消息
- 预留文件传输消息
- 拓扑配置消息

### `core-session`

负责：

- 设备发现
- 手工添加设备
- TLS 会话建立
- 设备配对
- 心跳
- 重连
- 连接状态管理

### `core-topology`

负责：

- 固定网格拓扑数据结构
- 待布局区和离线占位
- 拓扑合法性校验
- 拓扑持久化
- 拓扑热更新
- 切屏命中逻辑消费拓扑结果

### `core-input`

负责：

- 本地输入捕获
- 目标设备选择
- 输入事件转发
- 本地注入
- 后续扩展的鼠标灵敏度处理
- 后续扩展的滚轮平滑处理

### `core-clipboard`

负责：

- 剪贴板监听
- 文本格式抽象
- 剪贴板同步
- 循环同步抑制
- 为图片剪贴板扩展预留格式层

### `core-file-transfer`

负责：

- Post-MVP 文件元信息
- Post-MVP 分块传输
- Post-MVP 校验、确认、冲突处理
- Post-MVP 进度跟踪

### `app-desktop`

负责：

- Tauri 桌面壳
- 设置页面
- 拓扑拖拽 UI
- 配对确认
- 权限引导
- 日志与状态展示

## 5. 平台与进程设计

### Windows

MVP 计划采用 Rust 调用 Win32 API：

- 输入捕获
- 输入注入
- 光标控制
- 托盘管理

MVP 边界：

- 仅覆盖普通用户桌面会话
- 不覆盖 UAC、登录屏幕、安全桌面
- 不额外引入常驻系统服务

### macOS

MVP 计划采用 Rust 核心 + 少量系统桥接：

- 输入捕获
- 输入注入
- 权限检查与引导
- 菜单栏或后台常驻行为

MVP 边界：

- 仅覆盖用户登录后的普通桌面会话
- 需要处理 Accessibility、Input Monitoring 和 Secure Input 提示
- 不引入登录前特权助手

## 6. 网络与协议设计

采用局域网内星型主控拓扑：

- 一台主控机负责管理布局和输入流向
- 多台客户端负责接收控制和同步数据
- 所有设备之间通过受信连接建立会话

MVP 协议策略：

- 每个主控-客户端对使用一条 TLS/TCP 会话
- 在单条会话上承载版本化帧协议
- 协议内部按消息类别区分 `control`、`input`、`clipboard`、`diagnostic`
- 预留 `file_transfer` 消息类别，但不作为 MVP 验收范围
- 发送队列优先级为 `input > control > clipboard > diagnostic > file_transfer`
- 会话需要支持心跳和重连

## 7. 安全设计

默认安全模型：

- TLS 加密通信
- 每台设备有独立身份和本地证书
- 首次连接进行配对确认
- 配对时展示指纹进行用户确认
- 受信设备与证书指纹持久化
- 未配对设备直接拒绝

配对后能力授权：

- 输入共享
- 文本剪贴板共享
- Post-MVP 文件传输

补充约束：

- 证书变化视为新设备，需要重新配对
- 撤销信任后旧会话必须失效
- 不要求兼容 Deskflow 现有证书或配置文件

## 8. 拓扑设计

MVP 拓扑设计要求：

- UI 拖拽只负责生成布局操作
- 核心层负责校验合法性
- 核心层负责持久化
- 输入链只消费最终布局结果

MVP 拓扑模型约束：

- 采用固定网格模型，默认 3x3
- 一台设备只占用一个网格单元
- 支持主屏与多个副屏
- 支持上、下、左、右邻接
- 不支持重叠
- 不支持孤立节点
- 支持离线占位
- 支持热更新
- 不支持任意像素级自由矩形拼接

## 9. 性能与可观测性设计

关键原则：

- 输入事件使用紧凑协议
- 文本剪贴板同步不阻塞输入
- 后续的鼠标灵敏度与滚轮平滑在接收端本地处理
- UI 不进入高频路径
- 从 MVP 开始使用结构化日志和诊断上下文

MVP 关键性能关注点：

- 输入端到端延迟
- 高频鼠标移动连续性
- 拓扑热更新抖动
- CPU 和内存占用
